## 二、資料模型風險：扁平 `AnalysisEntry` 撐不住複雜零件

### 現況
`core/models.py` 裡 `AnalysisEntry` 有 18 個欄位，但全部是純量（一個 `length` / 一個 `width` / 一個 `material`）。任何「子結構」都被迫塞進 `remark`：

```python
# core/plate.py:59
remark = f"{plate_a}x{plate_b}x{plate_thickness}[{bolt_x}x{bolt_y}]_{bolt_hole}%{bolt_size}"
```

→ M42 plate 的螺孔分布、孔徑、配合螺栓規格全部變成一個無法解析的字串。

同樣的 anti-pattern 在 30+ 個 `type_XX.py` 反覆出現：
- `type_42.py:98` `result.entries[-1].remark = f"主梁 (立柱), H={h_mm}mm"`
- `type_43.py:106` `f"主梁, H={h_mm}+A={t43_data['A']}={main_len}mm"`
- `type_27.py:185` `f"Column, H={section_H}-15={effective_H}{l1l2_tag}"`

`remark` 在這些地方扮演了三種職責：
1. 工程師看的中文說明（OK）
2. 角色標籤（Column / Top beam / 斜撐 → 不該是字串）
3. 計算公式追溯（H=500-15=485 → 不該是字串）

### 風險
- **顯示**：UI 想把孔位資訊單獨一欄顯示，得反向 parse remark。
- **CAD/3D 自動化**：拿到 `LUG PLATE TYPE-C / 200×150×9` 後完全不知道孔位、孔徑、邊距、焊縫位置。
- **驗證**：單元測試只能比對字串，不能比對結構化欄位。
- **演化**：M42 標準改了螺孔位置，影響的是字串拼裝邏輯而不是資料層。

### 建議重構（最小可行）
把 `AnalysisEntry` 改成「核心欄位 + 子結構欄位」混合：

```python
@dataclass
class HolePattern:
    pattern: str            # "rect" / "circular" / "single"
    pitch_x: float = 0
    pitch_y: float = 0
    diameter: float = 0
    fastener_spec: str = ""  # '5/8"' / 'M16' ...
    count: int = 0

@dataclass
class GeometryHints:
    """讓 CAD 端能直接取，不需 parse remark"""
    role: str = ""          # "column" / "top_beam" / "lug" / "shim" / "wing"
    formula: str = ""       # "H + A" / "H - 15"  追溯
    holes: HolePattern | None = None
    notes_zh: str = ""      # 工程師看的中文，可選

@dataclass
class AnalysisEntry:
    # 既有欄位保留 ...
    geometry: GeometryHints = field(default_factory=GeometryHints)
    remark: str = ""        # 仍可保留作為簡述，但不再承擔結構資料
```

這樣 plate.py 裡的 remark 拼字就變成：

```python
entry.geometry = GeometryHints(
    role="base_plate",
    holes=HolePattern(pattern="rect", pitch_x=bolt_x, pitch_y=bolt_y,
                       diameter=bolt_hole, fastener_spec=bolt_size, count=4)
        if bolt_switch else None,
)
```

UI、Excel 匯出、未來 CAD/IFC 匯出都能直接對 `entry.geometry.holes` 操作。

## 三、命名/語意風險：沒有受控詞彙（你問「Lug_plate 是什麼」的根源）

### 現況
我直接搜了 `plate_name=` 的所有用法，光是「角板/lug plate」這一個概念就出現了：

| 寫法 | 出處 |
|------|------|
| `LUG_PLATE_C` | type_25.py, type_26.py |
| `LUG PLATE TYPE-C` | type_39.py, type_43.py, type_45.py, type_47.py |
| `Plate_a_有鑽孔` / `Plate_a_無鑽孔` | core/m42.py |
| `Plate_F` / `Plate_E` / `Plate_P` | type_07/10/12/13 |
| `Plate_WING` / `Plate_STOPPER` / `Plate_BASE` / `Plate_TOP` | type_14, 15 |
| `Plate_6t_Side` / `Plate_9t_Wing` | type_27 |
| `BASE PLATE` | type_41 |
| `COVER_PL` | type_12, 13 |
| `C/S SHIM` / `REIN. PAD` / `FLAT BAR` / `MEMBER C` | type_42, 51, 56, 61 |
| `SADDLE (120°)` | type_56 |

→ 同一個語意實體在不同 type 用了不同字串。`material_summary._classify_entry` 用 `entry.name.split("_")[0]` 來分組，遇到大寫字串、空格寫法、底線寫法時會分成不同 row。聚合表已經部分壞掉。

### 風險
- 採購聚合：`LUG_PLATE_C`、`LUG PLATE TYPE-C`、`Lug Plate Type C` 會被聚成 3 行。
- CAD 串接：人類看圖能猜，程式無法把字串對到圖庫零件。
- 國際化：英中混雜（`鋼板類` / `BASE PLATE`）讓 i18n 變不可能。
- 標準演化：你想把所有 lug plate 改名 `BRACKET_LUG`，得搜 30 幾個 type 檔逐字改。

### 建議重構：建立 ComponentRole 受控詞彙
既然 `support_ontology.json` 已經存在，順勢加一個 component role registry：

```python
# core/component_roles.py
class ComponentRole(str, Enum):
    BASE_PLATE              = "base_plate"
    LUG_PLATE               = "lug_plate"
    SHIM_PLATE              = "shim_plate"
    COVER_PLATE             = "cover_plate"
    WING_PLATE              = "wing_plate"
    STOPPER_PLATE           = "stopper_plate"
    SIDE_PLATE              = "side_plate"
    SADDLE_PLATE            = "saddle_plate"
    FLAT_BAR                = "flat_bar"
    REINFORCEMENT_PAD       = "reinforcement_pad"
    COLUMN                  = "column"
    TOP_BEAM                = "top_beam"
    DIAGONAL_BRACE          = "diagonal_brace"
    TRUNNION                = "trunnion"
    EXPANSION_BOLT          = "expansion_bolt"
    MACHINE_BOLT            = "machine_bolt"
    K_BOLT                  = "k_bolt"
    # ...
```

`AnalysisEntry` 再多一個 `role: ComponentRole | str`。
顯示用名稱 (`name`) 從 role + 圖面標籤推導，例如 `LUG_PLATE → "Lug Plate"`（en）/「角板」（zh）。
聚合 key 改用 `(role, spec, material)`，再也不會被字串大小寫或拼法影響。

CAD 端拿到 `role=lug_plate, geometry.holes=...` 就能直接查圖庫模型。

## 四、標準表格更新體驗：90% 是寫死的 Python（你的第四個痛點）

### 現況
- `core/config_loader.py` 已經支援 JSON 表格 + 自動 change_log + last_modified。
- `configs/type_01.json` 是唯一一個用了這個機制的 type。
- 但 `core/types/type_01.py` 實際上**沒有**透過 `get_type_table_as_dict("01")` 讀，仍依賴 `data/type01_table.py`（如果存在）或 hardcode 在 `.py`。
- 其他 60+ 個 type 全部把標準表寫成 Python dict（`data/type07_table.py`、`m42_table.py`、`m34_table.py`...）。

### 風險
- **法規/標準改值** = patch 一堆 `.py`，過程中可能誤改邏輯。
- 沒有 schema 驗證，標準表筆誤（單位寫錯、欄位漏填）只能靠執行時報錯。
- 沒有版本欄位，不知道目前用的是 IEC 哪一版。
- Code review 時「資料變更」和「邏輯變更」混在同一個 PR 裡。

### 建議重構：兩個小步驟
1. **把 data/typeXX_table.py 全部搬到 configs/typeXX.json**
   - `config_loader` 已經寫好，只缺一個 migration script。
   - 加一個 `configs/_schema/typeXX.schema.json`（JSON Schema），CI 跑 `jsonschema` 驗證。
   - 加 `configs/_schema/standard_version.json`：記錄目前依據哪個版本的圖面/標準。
2. **新增 `data/__bridge__.py`** 暫時保留舊 import 介面：
   ```python
   def get_type07_data(line_size):
       return get_type_table_as_dict("07").get(int(line_size))
   ```
   讓既有 `type_07.py` 不需要動，但底層改吃 JSON。

實際做完之後，標準改值的流程會變成：「改一個 JSON、跑 schema 驗證、commit。」

## 五、計算邏輯結構：60+ 個 calculate 各自為政（你的第三個痛點）

### 現況
每個 `type_XX.py` 都長得很像，但又不完全一樣：
- 解析 `get_part(fullstring, n)` 的 boilerplate 重複
- 邊界檢查（H_max / L_max / 字母白名單）寫法各異
- BOM 建立順序混雜業務邏輯與 `result.entries[-1].remark = ...` 的 side-effect

例如 type_42.py 有 5 步、type_43.py 有 7 步、type_27.py 有兩個 if/else 分支各自呼叫 add_*。**「整修一個 type 的計算」其實是「同時改解析、邊界、查表、組裝、備註」五件事**——這就是你說的「無力感」。

### 風險
- 一個 bug fix 動到 `add_steel_section_entry` 的順序，會打破 BOM 編號。
- 加新欄位（例如「焊縫類型」）要改 60 個檔案。
- 行為差異難以歸納（「這個 type 是用 H+A 還是 H-15？」沒地方一覽）。

### 建議重構：把 calculate 拆成三個層次

```
type_XX.py 應該只負責「定義 spec」，而不是「執行流程」：

class Type42Spec(TypeSpec):
    designation_pattern = "42-{line_size}B-{member}-{H} {fig}"
    fields = [LineSize(2), MemberCode(3), HFig(4)]
    constraints = [
        H_max_lookup_from_member,
        member_in_supported_members,
    ]
    bom = [
        BomItem(role="column",         length=lambda ctx: ctx.h),
        BomItem(role="diagonal_brace", length=lambda ctx: ctx.g),
        BomItem(role="trunnion",       spec_lookup="pipe_data.trunnion"),
        BomItem(role="shim",           dims=("C", "D", 6)),
        BomItem(role="machine_bolt",   spec='3/4"x50', qty=2),
    ]
```

執行邏輯（解析 → 邊界檢查 → 查表 → 組 BOM）由 framework 完成。
寫新 type 時人腦只在 spec 上打轉，不需要記住 `result.add_entry` / `result.entries[-1].remark` 這些 side-effect。

過渡期可以**保留 calculate() 介面**，內部改成 `return TYPE42_SPEC.execute(fullstring, ...)`。新舊共存。

## 六、其他結構性風險（簡列）

1. **`material_summary._classify_entry` 用字串前綴判斷分類**——一旦 `name` 改成「Top Beam」就會被誤分到 piece。應該由 `entry.role` 直接決定。
2. **`m42.py` 的字母 → 動作對應表是 25 條 lambda**——加新字母要動 actions dict 跟 perform_action_by_letter 兩處，且各 lambda 寫死了「呼叫順序」。建議改成資料驅動：
   ```json
   "B": [
       {"action": "plate", "type": "a"},
       {"action": "plate", "type": "d"},
       {"action": "bolt",  "qty": 4}
   ]
   ```
3. **`pdf_export.py` 把 `description` / `remark` 等資料硬塞進固定 11 欄**——改了 entry 結構就要連動 export，缺乏列定義介面。
4. **`assets/Type/*.pdf` 沒有對應的版次/標準引用 metadata**——改版時不知道哪些 PDF 已經過期。建議在 `type_catalog.json` 每個 type 下增加 `source_drawings: [{file, drawing_no, rev, date}]`。
5. **沒看到 `tests/` 目錄**（只看到 `test_material_cutting.py` 一支）——重構時沒有回歸保護。建議至少對每個 type 建一筆 golden case。

## 七、優先順序建議

如果只能挑兩件事先做，會是：

1. **先把 `AnalysisEntry` 加上 `role` + `geometry.holes`**（向後相容，舊欄位不動）
   ↓ 立刻解決：M42 螺孔資訊不再進 remark；採購聚合可以靠 role；UI 可以多一欄顯示孔位。
2. **把 `data/typeXX_table.py` 遷到 `configs/typeXX.json`**
   ↓ 立刻解決：未來標準改版只動資料；可以加 schema 驗證；可以加 standard version。

這兩件做完，「無力感」會明顯下降——剩下的 calculate spec 化、命名 enum 化都可以分批漸進，**不需要 big-bang refactor**。

| 階段 | 變更 | 風險 | 收益 |
|------|------|------|------|
| Phase 0 | 新增 `role: str = ""`、`geometry: dict = {}` 兩個 optional 欄位 | 0（純加） | 開始能在新 type 試驗結構化 |
